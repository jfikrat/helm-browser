## Helm — Real Chrome Validation Test Plan

Highest-risk areas first. All tests assume the MCP server is running and the extension is loaded.

### 1. Persistent Debugger — Grace-Period Race

**Affected code**: `acquireDebugger` (commands.js:124), `releaseDebugger` (commands.js:150), `DEBUGGER_GRACE_PERIOD_MS = 5000` (commands.js:23)

**Why high-value**: The race is structurally real. When the 5 s grace timer fires, the async function clears `session.detachTimer = null` *before* checking `refCount`. If a new `acquireDebugger` runs in that gap — timer has already nulled the field so the `clearTimeout` guard in `acquireDebugger:127` doesn't fire, refCount is incremented, `session.attached === true` returns immediately — then the timer's `detach()` completes and `cleanupDebuggerSession` overwrites refCount to 0 and deletes the session. The next CDP send on the returned `dbg` handle fails silently.

**Manual recipe**:
1. Call `screenshot` to attach the debugger
2. Wait exactly 5 s (don't interact)
3. Immediately fire a second `screenshot` *while* watching DevTools → Extensions → Service Worker logs for `[Helm] Debugger detached`
4. Repeat 10× — at least 1–2 runs should hit the race

**Expected signal**:
- Success: screenshot returns image every time
- Failure: `{error: "Debugger session for tab X was cleaned up during attach"}` or `{error: "Cannot access a chrome:// URL"}` appearing randomly on the same page, or a blank/null `image` field

### 2. Full-Page Screenshot — Viewport Override Leak

**Affected code**: commands.js:584-629, `Emulation.setDeviceMetricsOverride` at line 598, cleanup at line 619

**Why high-value**: If `Page.captureScreenshot` throws (network error, about:blank, very large page), the inner `finally` does attempt `Emulation.clearDeviceMetricsOverride` — but that call is swallowed silently (line 619 catch: `{}`). If the debugger is already in a bad state at cleanup time, the viewport stays inflated. Every subsequent viewport screenshot on this tab then captures a wrong-sized image.

**Manual recipe**:
1. Call `browser_screenshot` with `fullPage: true` on a tall page (e.g., a long Wikipedia article)
2. Immediately call `browser_screenshot` *without* `fullPage` on the same tab
3. Check the returned image dimensions in both calls
4. As a stress test: interrupt full-page capture by navigating mid-capture; repeat step 2

**Expected signal**:
- Success: second screenshot dimensions match the actual viewport (~1280×800 or configured size)
- Failure: second screenshot is unusually tall (hundreds or thousands of px height), showing scrolled content or empty space below the fold

### 3. `doubleClick` — Wrong CDP Click Sequence

**Affected code**: `dispatchMouseSequence` (commands.js:236), called at line 1189 with `{button: 'left', clickCount: 2, repeatCount: 2}`

**Why high-value**: A CDP double-click requires the *first* press to have `clickCount: 1` and the second to have `clickCount: 2`. The current loop sends both presses with `clickCount: 2`. Sites that detect double-click via `event.detail === 2` or `dblclick` DOM event may still work (Chrome maps the sequence), but sites using `mousedown`/`mouseup` `event.detail` directly will receive two `detail=2` events instead of a `1→2` progression, breaking custom drag-select, inline-edit activation, and canvas interactions.

**Manual recipe**:
1. Open a page with an editable element that uses double-click to select a word (any `contenteditable` div, Google Docs, CodeMirror editor)
2. Call `browser_double_click` on a word
3. Check if the word is selected after the call

**Expected signal**:
- Success: word is visually selected
- Failure: cursor placed but no word selected, or only the second click registered, or selection jumps unexpectedly

### 4. `pressKeys` — Modifier Keys Stuck on Mid-Sequence Error

**Affected code**: commands.js:1944 (modifier keyUp loop is inside `try`, not `finally`)

**Why high-value**: If the non-modifier key dispatch (line 1930+) throws an exception — e.g., the element has scrolled off-screen or the tab navigated — the `for` loop at line 1944 that releases held modifiers (`Ctrl`, `Shift`, `Alt`) is skipped. The CDP session still has those modifiers "held" from the `keyDown` events. Every subsequent `click` or `type` dispatched via CDP on that tab picks up the phantom modifiers (Ctrl+click opens new tab, Shift+click extends selection, etc.).

**Manual recipe**:
1. Call `browser_press_keys` with `keys: ["Control", "a"]` on a page
2. Navigate the page away mid-call (open DevTools console, call `chrome.tabs.update(tabId, {url: 'about:blank'})` during the key sequence)
3. Navigate back to the original page and call `browser_click` on any element
4. Observe if Ctrl is still "held" (e.g., click opens a new tab instead of following link)

Simpler test: call `browser_press_keys` with an invalid selector, then `browser_click` — check if the click behaves as a modifier-click.

**Expected signal**:
- Success: click after failed pressKeys behaves as a normal click
- Failure: link opens in a new tab, text gets selected instead of clicked, or devtools shows modifier keys in the dispatched event

### 5. `waitForFunction` — Syntax Errors Time Out Silently

**Affected code**: commands.js:2453, `Runtime.evaluate` call — no check for `result.exceptionDetails` unlike `executeScript` at line 2429

**Why high-value**: A bad expression (unclosed string, undefined variable) causes the evaluate to return an exception result. `result.result.value` is `undefined`, which is `!== true`, so the loop runs for the full `clampedTimeout` (up to 60 s) and returns `{success: false, error: 'Timeout'}`. The caller has no idea the expression was malformed.

**Manual recipe**:
1. Call `browser_wait_for_function` with `expression: "document.querySelector('"` (deliberately broken syntax)
2. Observe whether it returns immediately with an error or hangs for the full timeout
3. Separately call `browser_wait_for_function` with `expression: "undeclaredVar === 42"` and observe behavior

**Expected signal**:
- Success (desired): returns `{success: false, error: "SyntaxError: ..."}`  immediately
- Failure (current behavior): hangs for `timeout` ms then returns generic timeout error

### 6. Screenshot — Concurrent Debugger Sharing During `waitForFunction`

**Affected code**: `waitForFunction` (commands.js:2450) holds `acquireDebugger` for the entire polling duration; `screenshot` in unfocused-window path also calls `acquireDebugger` (line 585 or line 644)

**Why high-value**: Both commands share the same `debuggerSessions` entry via refCount. `waitForFunction` calls `Runtime.evaluate` in a loop. If `screenshot` fires concurrently and calls `Page.enable` or `Emulation.setDeviceMetricsOverride`, CDP domain state changes affect all concurrent commands sharing the session. `Emulation.setDeviceMetricsOverride` during a `waitForFunction` poll could cause subsequent `Runtime.evaluate` to run against a resized viewport, making DOM queries return different values. The result returns `success: true` at the wrong time.

**Manual recipe**:
1. Start `browser_wait_for_function` with a 10 s timeout expression
2. Immediately issue `browser_screenshot` with `fullPage: true`
3. Check the `waitForFunction` result — did it succeed/fail at the right time?
4. Check the screenshot image dimensions

**Expected signal**:
- Success: both complete independently, screenshot has correct dimensions
- Failure: `waitForFunction` resolves early or at wrong time; screenshot image height doesn't match page

### 7. `clearCookies` — Domain Cookie Removal URL Construction

**Affected code**: `buildCookieRemovalUrl` (handlers.js:201), called from `clearCookies` loop (handlers.js:269)

**Why high-value**: For cookies scoped to `.example.com` (leading dot), `buildCookieRemovalUrl` strips the dot and constructs `https://example.com/`. But `chrome.cookies.remove` needs a URL matching the actual cookie's effective URL. Cookies with `domain=.example.com` set on `subdomain.example.com` might not be removed by a URL pointing at the apex domain. Silent non-removal: `removedCount` stays at 0, but the return is still `{success: true, removedCount: 0}`.

**Manual recipe**:
1. Use `browser_set_cookie` to create a cookie with `domain: ".example.com"` on a real site
2. Call `browser_clear_cookies` with the site URL
3. Immediately call `browser_get_cookies` and verify the cookie is gone

**Expected signal**:
- Success: cookie absent after clear
- Failure: `removedCount: 0` on return but cookie still present in `get_cookies`

### 8. Service Worker Restart — Debugger State Loss

**Affected code**: `debuggerSessions` Map in state.js:26 (in-memory, lost on SW termination); `handlers.js:65` `onDetach` listener re-registered per SW lifecycle

**Why high-value**: Chrome terminates idle service workers after 30 s. After restart, `debuggerSessions` is empty (`attached = false`). `acquireDebugger` calls `chrome.debugger.attach`, but if Chrome still has the debugger attached from the previous SW instance, this throws `"Already attached to this target"`. There's no fallback to detach-then-reattach. Every tool that requires the debugger (screenshot, pressKeys, waitForFunction, full-page screenshot, waitForNetworkIdle) will fail until the tab is reloaded.

**Manual recipe**:
1. Take a screenshot (attaches debugger, starts 5 s timer)
2. Wait 35+ seconds without any MCP calls (SW terminates)
3. Take another screenshot immediately

**Expected signal**:
- Success: screenshot returns an image
- Failure: `{error: "Cannot attach to this target"}` or similar, on every debugger-dependent tool

## Observability Blind Spots

- **`acquireDebugger` failures don't surface the tab state**: When the attach race occurs, the caller gets a generic error. There's no diagnostic field indicating whether the failure was "already attached by another actor" vs "session cleaned up mid-attach" vs "tab closed". Add tag to the error to distinguish.

- **`waitForFunction` expression errors are absorbed**: No way to distinguish timeout-due-to-expression-never-true from timeout-due-to-syntax-error. Both return identical `{success: false, error: 'Timeout'}`.

- **Full-page screenshot leaves no trace of viewport override failure**: The `Emulation.clearDeviceMetricsOverride` catch at line 619 is completely silent — not even a `console.warn`. If cleanup fails, the next screenshot on that tab will be wrong with no indication why.

- **`pressKeys` modifier leak**: No observable state surface — the CDP modifier mask is internal to the debugger session. The only way to detect it is behavioral (subsequent clicks act as modifier-clicks). There's no health-check command to inspect current modifier state.

- **SW restart vs. debugger still attached**: The `chrome.debugger.onDetach` listener correctly cleans up state when Chrome detaches the debugger. But it can only fire while the SW is alive. If the SW dies while a debugger is attached, the onDetach never fires for that instance. The new SW starts clean with no record of the attached debugger.
